# 全球AI·科技·财经动态 — 后端架构与开发路线图

> 目标：自动化采集 → 结构化存储 → 定时生成静态页面 → 一键部署
>
> 核心原则：简洁、低维护成本、适合中国网络环境

---

## 一、系统架构总览

```
┌─────────────────────────────────────────────────────────┐
│                    定时触发器 (Cron)                      │
│             每 30 分钟 / 1 小时 / 6 小时                 │
└────────────┬───────────────────────────────┬────────────┘
             │                               │
┌────────────▼──────────────┐  ┌────────────▼──────────────┐
│       采集层               │  │       生成层               │
│                           │  │                           │
│  RSS 源 → feedparser      │  │  读取 JSON 数据           │
│  API 源 → httpx           │  │  Jinja2 模板渲染           │
│  去重 & 结构化             │  │  生成 HTML 页面            │
│  → 写入 JSON 数据文件      │  │  → 输出到 dist/ 目录      │
└────────────────────────────┘  └───────────────────────────┘
                                        │
                               ┌────────▼────────┐
                               │    部署层        │
                               │                 │
                               │  dist/ 目录      │
                               │  → GitHub Pages  │
                               │  → Vercel        │
                               │  → 任意静态托管   │
                               └──────────────────┘
```

### 为什么选择「静态站点生成」方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **静态站点生成** ✅ | 免费部署（GitHub Pages/Netlify）、CDN加速、零服务器运维、低成本 | 非实时（本文方案每30分钟更新一次） |
| 动态后端（FastAPI+DB） | 实时更新、可加用户功能 | 需要VPS、运维成本高、国内部署麻烦 |
| 纯前端JS动态加载 | 部署简单 | SEO差、跨域问题、需要代理 |

**结论**：静态站点生成器最适合这个项目——新闻聚合不需要实时推送，30分钟更新一次完全够用。

## 二、数据模型

### 2.1 新闻条目 (NewsItem)

```python
@dataclass
class NewsItem:
    id: str              # 唯一ID (URL的hash或uuid)
    title: str           # 标题
    summary: str         # 摘要
    url: str             # 原文链接
    source_name: str     # 来源名称 (如 "36氪", "Bloomberg")
    source_icon: str     # 来源图标 (单字母)
    source_color: str    # 来源品牌色 (HEX)
    category: str        # 分类: AI | TECH | FINANCE | VIDEO
    published_at: str    # 发布时间 (ISO格式)
    fetched_at: str      # 采集时间 (ISO格式)
    thumbnail_url: str   # 缩略图URL (可选)
```

### 2.2 数据存储结构

```
data/
├── sources.yaml          # 数据源配置
├── news/
│   ├── AI/
│   │   ├── 2026-07-16.json    # 每天一个文件
│   │   └── 2026-07-15.json
│   ├── TECH/
│   │   └── ...
│   ├── FINANCE/
│   │   └── ...
│   └── VIDEO/
│       └── ...
├── index.json            # 汇总索引（所有分类的最新50条）
└── meta.json             # 元数据（各分类数量、更新时间等）
```

## 七、关键设计决策

### 7.1 去重策略

- 以 `URL` 的 SHA256 hash 为唯一 ID
- 存储时检查 ID 是否已存在，避免重复
- 同一条新闻在不同数据源中出现 → 保留最早采集到的版本

### 7.2 分类规则

- 每个数据源固定映射到一个分类（在 `sources.yaml` 中配置）
- 示例：`36氪` → AI/TECH/FINANCE 混编，需要根据标题关键词二次分类
- 关键词规则：包含 "GPT"/"大模型"/"AI"/"算法" → AI；"芯片"/"手机"/"汽车" → TECH；"股市"/"美元"/"利率" → FINANCE

### 7.3 增量更新

- 每次采集只获取最新 N 条（RSS 默认 20-50 条）
- 存储时合并到当天文件
- 首页展示各分类最新的 50 条
- 二级页通过加载更多展示历史数据（翻页访问历史文件）

### 7.4 代理配置

- 国际源使用系统代理 `HTTP_PROXY=http://127.0.0.1:7897`（已有配置）
- 国内源直连

## 八、风险与备选方案

| 风险 | 影响 | 备选方案 |
|------|------|----------|
| RSS 源失效 | 数据变少 | 增加更多备用源，监控并自动切换 |
| 国际 API 被墙 | 无法获取数据 | 纯国内源方案，或使用 RSSHub 反向代理 |
| GitHub Actions 中国访问慢 | 部署延迟 | 使用 Gitee Pages 或国内 VPS |
| 存储 JSON 文件膨胀 | 占用空间 | 定期清理 30 天前的旧文件 |
| 模板与数据分离后样式偏差 | 页面显示异常 | 构建完成后自动截图对比验证 |
